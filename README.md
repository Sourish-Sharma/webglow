# Automated Website Generation and Deployment Pipeline

## Overview

This project implements an automated pipeline that converts structured restaurant data into deployable frontend websites.

It is designed as a downstream system for the scraper available at:
https://github.com/Sourish-Sharma/gmap-scrapper

The upstream project collects raw data from Google Maps and processes it into structured JSON. This pipeline consumes that data, generates complete frontend code, and deploys each site automatically.

---

## System Architecture

The pipeline operates in three stages:

### Phase 1: Data Enrichment

* Reads structured JSON data generated by the scraper
* Uses a language model to extract:

  * Brand identity
  * Target audience
  * Value proposition
  * Menu highlights
  * Operational metadata
* Enforces a fixed JSON schema to ensure consistency

### Phase 2: Code Generation

* Converts enriched data into constrained prompts
* Generates:

  * `index.html`
  * `styles.css`
  * `script.js`
* Applies validation before saving:

  * Ensures required keys exist
  * Checks HTML structure (`<body>`, closing tags)
  * Rejects incomplete or truncated outputs

### Phase 3: Deployment

* Initializes a Git repository for each generated site
* Creates a remote repository on GitHub
* Deploys the site using Vercel

Each generated website is independent and can be managed separately.

---

## Integration with Scraper

This project depends on the output of:
https://github.com/Sourish-Sharma/gmap-scrapper

Workflow:

1. Scraper collects listings and reviews
2. Data is converted into structured JSON
3. This pipeline processes the JSON
4. Websites are generated and deployed

This separation keeps data collection and application generation independent.

## Security Considerations

The implementation includes several measures to reduce risk during automated execution.

### API Key Handling

* API keys are not stored in source code
* All credentials are loaded through environment variables
* No fallback keys are included

### Network Behavior

* HTTP client configuration disables environment-based proxy inheritance
* Prevents unintended routing through local proxy settings

### Input Validation

* Model outputs are cleaned to remove formatting artifacts
* JSON responses are parsed and validated before use
* Required schema fields are checked before processing

### Output Validation

* Generated HTML must meet minimum size requirements
* Structural checks ensure valid HTML before writing to disk
* Invalid outputs are rejected and retried

### Execution Control

* Concurrency is limited using semaphores
* Requests are spaced to respect API rate limits
* Retry logic uses exponential backoff with jitter

### State Management

* Completed records are tracked using checkpoint files
* Failed runs can resume without reprocessing completed items

### Repository Safety

* Deployment script checks for existing repositories
* `.gitignore` is required before pushing code
* Default visibility is private unless explicitly set otherwise
